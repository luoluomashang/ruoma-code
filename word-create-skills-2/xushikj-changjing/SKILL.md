# 叙事空间创作系统 - 场景模块

## 概述

执行宏观十二工作流的步骤 8-9：场景清单编制与场景规划。将四页大纲拆解为具体的场景单元，为每个场景规划视点人物、冲突、目标等必要信息，并标注敏感度标签，为逐章创作做好全部准备。

## 外部工具调用规范 (MCP Services)
在规划场景前，可以调用以下工具：
**`Websearch` (网络搜索)**：在规划场景时，用于搜索合适的场景节奏与细腻度，保证场景规划松弛有度。搜索词必须精准，提取信息后必须提炼。

你已接入 MCP 服务，拥有强大的逻辑推演能力。在规划核心冲突场景时，你**必须**使用以下工具：
- **`sequential-thinking` (序列化深度思考)**：用于在生成场景规划前，在后台推演反派的杀局逻辑和主角的破局链路，确保反派智商在线、主角赢得漂亮且符合逻辑。

## 执行流程

### 步骤8: 场景清单

#### 8.1 场景拆解

从四页大纲中拆解出所有场景，拆解前调用 `Websearch` 工具进行网络搜索，保证场景的节奏和细腻度，保证场景节奏符合预期且松弛有度，每个场景必须包含矛盾冲突，搜索词必须精准，提取信息后必须提炼，提炼后调用 `sequential-thinking` 工具按照节奏识别每一个独立的场景单元。

场景清单格式：

```markdown
# 场景清单

## 第一幕

| 场景编号 | 场景标题 | 所在章节(预估) | 视点人物 | 核心冲突 | 敏感度 |
|----------|----------|---------------|----------|----------|--------|
| S001 | 重生之日 | 第1章 | 主角 | 发现自己重生，面对旧日困境 | GREEN |
| S002 | 初遇宿敌 | 第1-2章 | 主角 | 与前世害死自己的人再次相遇 | YELLOW |
| ... | ... | ... | ... | ... | ... |

## 第二幕
...

## 第三幕
...
```

#### 8.2 场景校验

- 每个场景是否都有明确的矛盾冲突
- 场景的顺序是否符合三幕式结构
- 信息差释放是否按层级分布在不同场景中
- 钩子九连环是否贯穿全部场景
- 涉及的角色和地点是否与知识库一致

#### 8.3 敏感度标注

根据 `safety_guard.yaml` 的敏感度标签系统，为每个场景标注敏感度：

| 标签 | 描述 | 适用场景 |
|------|------|----------|
| GREEN | 日常、对话、风景、修炼 | 普通叙事场景 |
| YELLOW | 战斗、心理博弈、情感冲突 | 中等强度冲突场景 |
| RED | 亲密互动、暴力细节、黑暗主题 | 高强度描写场景 |

### 步骤9: 场景规划

为每个场景编写详细规划，保存为独立文件。

#### 场景规划模板

```markdown
# 场景 S001: [场景标题]

## 基本信息
- **视点人物**: [角色名]
- **所在地点**: [地点名（对应知识库 loc_xxx）]
- **时间**: [故事内时间]
- **敏感度**: [GREEN/YELLOW/RED]
- **预估章节**: [第X章]

## 主动场景
- **目标**: [视点人物在本场景中想要达成什么]
- **冲突**: [阻碍目标达成的力量/事件/角色]
- **挫折**: [冲突导致的具体后果]

## 被动场景
- **反应**: [视点人物对挫折的情感反应]
- **困境**: [挫折后面临的两难选择]
- **决定**: [视点人物最终做出的选择]

## 涉及角色
- [角色1（char_xxx）]: [在本场景中的作用]
- [角色2（char_xxx）]: [在本场景中的作用]

## 涉及物品/道具
- [物品1（item_xxx）]: [在本场景中的使用方式]

## 法则应用
- **极限铺垫**: [本场景如何铺垫后续爽点]
- **期待感**: [本场景设置了什么钩子]
- **连锁震惊**: [本场景触发了几层震惊反应]

## 章节末钩子（如本场景为章节结尾）
- [钩子描述]

## 双保险标注（YELLOW/RED 场景）
- **声明级别**: [L1/L2/L3]
- **描写类型标签**: [战斗/亲密/心理博弈/暴力/黑暗主题]
```

#### 9.1 强制逻辑推演 (Sequential-thinking)
在输出具体的场景文件前，对于任何包含核心冲突（特别是 YELLOW 和 RED 级别）的场景，你**必须**先调用 `sequential-thinking` 工具进行后台推演。

**推演思考链（Thought Chain）要求：**
1. **反派视角的“完美必杀”**：反派的真实目的是什么？他布下了一个怎样看似天衣无缝、毫无破绽的局？（必须体现出反派的高智商、狠辣或严密的权谋逻辑，绝对禁止降智行为）。
2. **读者的“绝望感”**：这个局在旁观者看来为什么是必死无疑的？（建立极致的压迫感）。
3. **主角的“信息差”盘点**：主角目前手头有哪些底牌？（特定的金手指能力、隐藏的战力、或者某个反派不知道的情报）。
4. **高智商破局点**：主角如何利用这个信息差，像庖丁解牛一样找到反派完美杀局中的致命盲区，并给予降维打击？
5. **错位震惊**：当主角掀桌子时，反派的信仰和骄傲将如何崩塌？

*注意：只有当 `sequential-thinking` 推演出了逻辑自洽、且能带来极大爽感的闭环后，才允许进入 9.2 步生成文件。*

#### 9.2 一致性交叉检查

规划完成后，进行全局校验：

- 角色的出场频率是否合理（主角占比最高）
- 地点切换是否自然（不存在瞬移）
- 物品使用是否与知识库状态一致
- 敏感度标签是否准确
- 钩子是否环环相扣
- 信息差释放节奏是否合理

## 双保险机制说明

对于标注为 YELLOW 或 RED 的场景，在后续逐章创作时将自动触发双保险机制：

### 保险一：前置专业上下文声明

根据场景敏感度，在创作前注入对应级别的声明：
- GREEN 场景：仅注入 L1（项目级声明）
- YELLOW 场景：注入 L1 + L2（步骤级声明）
- RED 场景：注入 L1 + L2 + L3（段落级声明）

### 保险二：输出完整性验证

创作完成后检查：
- HC1: 字数达标率是否 >= 70%
- HC2: 是否存在截断
- HC3: 是否存在内容降级（如「此处省略」）
- HC4: 是否出现拒绝信号
- HC5: 是否被注入安全提示

检查结果分为 PASS / WARN / HALT 三级，HALT 时需要用户介入决策。

## 注意事项

- 场景清单是创作的「施工蓝图」，必须足够细致
- 每个场景的冲突必须服务于整体叙事，不能为了冲突而冲突
- 敏感度标注直接影响后续创作的声明注入策略，需准确
- 场景编号（S001、S002...）将在后续模块中被引用

## 完成后

告知用户场景规划已全部完成，展示统计信息（总场景数、各敏感度场景数量、预估总章节数），询问是否进入下一步：写作模块（xushikj-xiezuo）。
